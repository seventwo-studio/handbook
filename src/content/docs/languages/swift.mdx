## Overview

Swift is used for SevenTwo Studio's native iOS development, exclusively with SwiftUI for user interfaces. We target **Swift 6.2** and **iOS 18** as the minimum deployment target. Our Swift standards emphasize safety, modern concurrency with strict concurrency checking, protocol-oriented programming, and zero third-party dependencies wherever possible.

## Optionals and Safety

Swift's optional system is one of its strongest features for preventing null pointer errors. We avoid force unwrapping (`!`) in all cases, as it can cause runtime crashes. Instead, we use optional binding, optional chaining, and guard statements to handle optionals safely.

Prefer `guard let` over `if let` when the unwrapped value is needed in the remaining scope â€” it keeps the happy path unindented and exits early on failure. Use `if let` for short, self-contained checks where the unwrapped value is only needed inside the branch.

```swift
// guard for early exit â€” unwrapped value available below
guard let user = session.currentUser else { return }
updateProfile(for: user)

// if let for scoped checks
if let cached = cache[key] {
    return cached
}
```

<Aside type="caution">
  Treat implicitly unwrapped optionals (`String!`) the same as force unwraps â€” avoid them outside of `@IBOutlet` (which we don't use with SwiftUI anyway).
</Aside>

## Concurrency

We use Swift's structured concurrency model exclusively: `async`/`await`, `TaskGroup`, and actors. Completion handlers and Combine publishers are legacy patterns â€” do not introduce them in new code.

### Actors and Sendable

Use actors to encapsulate mutable shared state. Mark types that cross concurrency boundaries as `Sendable` and enable strict concurrency checking (`-strict-concurrency=complete`) project-wide.

```swift
actor ImageCache {
    private var store: [URL: Image] = [:]

    func image(for url: URL) -> Image? {
        store[url]
    }

    func set(_ image: Image, for url: URL) {
        store[url] = image
    }
}
```

Use `@MainActor` for anything that touches UI state â€” ViewModels, navigation state, and any property read by SwiftUI views.

### Task management

Cancel tasks when they're no longer needed. Prefer `task` view modifier for work tied to a view's lifecycle, and `.task(id:)` when the work should restart when a value changes.

```swift
struct PhotoView: View {
    let photoID: String

    var body: some View {
        content
            .task(id: photoID) {
                await loadPhoto(photoID)
            }
    }
}
```

## Protocol-Oriented Programming

Use protocols to define contracts for testability and flexibility. Prefer protocol extensions for shared default behavior over base classes. Keep protocols focused â€” split large protocols into smaller, composable ones.

```swift
protocol PhotoLibrary: Sendable {
    func photos(in album: Album) async throws -> [Photo]
    func save(_ photo: Photo) async throws
}

// Default implementation via extension
extension PhotoLibrary {
    func firstPhoto(in album: Album) async throws -> Photo? {
        try await photos(in: album).first
    }
}
```

## Architecture

We follow **MVVM** with SwiftUI. Views are declarative and contain no business logic. ViewModels are `@Observable` classes annotated with `@MainActor` that own the state and drive the view.

```swift
@MainActor
@Observable
final class AlbumViewModel {
    private(set) var albums: [Album] = []
    private(set) var isLoading = false
    var error: Error?

    private let library: PhotoLibrary

    init(library: PhotoLibrary) {
        self.library = library
    }

    func load() async {
        isLoading = true
        defer { isLoading = false }
        do {
            albums = try await library.fetchAlbums()
        } catch {
            self.error = error
        }
    }
}
```

<Aside>
  We use the Observation framework (`@Observable`) instead of the older `ObservableObject` / `@Published` / `@StateObject` / `@ObservedObject` pattern. The Observation framework is more performant (fine-grained tracking) and requires less boilerplate.
</Aside>

### Dependency injection

Pass dependencies through initializers for ViewModels. For values that need to flow through the view hierarchy without explicit passing, use SwiftUI's `@Environment` with custom `EnvironmentKey` conformances.

```swift
struct PhotoLibraryKey: EnvironmentKey {
    static let defaultValue: PhotoLibrary = LivePhotoLibrary()
}

extension EnvironmentValues {
    var photoLibrary: PhotoLibrary {
        get { self[PhotoLibraryKey.self] }
        set { self[PhotoLibraryKey.self] = newValue }
    }
}
```

## SwiftUI Patterns

### Navigation

Use `NavigationStack` with value-based navigation for type-safe, programmatic control. Keep navigation state in a dedicated model so it can be tested and persisted.

```swift
@MainActor
@Observable
final class Router {
    var path = NavigationPath()

    func navigate(to destination: Destination) {
        path.append(destination)
    }

    func popToRoot() {
        path = NavigationPath()
    }
}
```

### Sheets and alerts

Drive presentations from optional state or boolean bindings â€” never call imperative presentation methods.

```swift
struct AlbumView: View {
    @State private var selectedPhoto: Photo?

    var body: some View {
        albumGrid
            .sheet(item: $selectedPhoto) { photo in
                PhotoDetailView(photo: photo)
            }
    }
}
```

### View composition

Extract reusable pieces into smaller views or `ViewModifier` conformances. Keep `body` readable â€” if it exceeds ~40 lines, split it.

```swift
struct CardModifier: ViewModifier {
    func body(content: Content) -> some View {
        content
            .padding()
            .background(.regularMaterial)
            .clipShape(RoundedRectangle(cornerRadius: 12))
    }
}

extension View {
    func card() -> some View {
        modifier(CardModifier())
    }
}
```

## Data Persistence

### SwiftData

SwiftData is our primary persistence layer. Define models with the `@Model` macro and use `ModelContainer` / `ModelContext` from the environment.

```swift
@Model
final class Project {
    var name: String
    var createdAt: Date
    @Relationship(deleteRule: .cascade)
    var tasks: [ProjectTask] = []

    init(name: String) {
        self.name = name
        self.createdAt = .now
    }
}
```

Use `@Query` in views for reactive, filtered, sorted data:

```swift
struct ProjectListView: View {
    @Query(sort: \Project.createdAt, order: .reverse)
    private var projects: [Project]

    var body: some View {
        List(projects) { project in
            ProjectRow(project: project)
        }
    }
}
```

### SQLite (via direct access)

For performance-critical queries, complex joins, or full-text search that SwiftData can't express well, drop down to SQLite directly. Wrap access behind a protocol so the persistence layer stays swappable.

## Networking

Use the native `URLSession` for all networking. Combine it with `async`/`await` and `JSONDecoder` for clean request/response flows.

```swift
struct APIClient {
    private let session: URLSession
    private let decoder: JSONDecoder

    func fetch<T: Decodable>(_ request: URLRequest) async throws -> T {
        let (data, response) = try await session.data(for: request)
        guard let http = response as? HTTPURLResponse,
              (200..<300).contains(http.statusCode) else {
            throw APIError.badResponse
        }
        return try decoder.decode(T.self, from: data)
    }
}
```

No third-party HTTP libraries â€” URLSession covers everything we need.

## Testing

We use **Swift Testing** (`@Test`, `#expect`, `#require`) for all tests. Do not use XCTest for new test code.

```swift
@Test("Fetching albums returns sorted results")
func fetchAlbums() async throws {
    let library = MockPhotoLibrary(albums: [.fixture(name: "B"), .fixture(name: "A")])
    let vm = AlbumViewModel(library: library)
    await vm.load()
    #expect(vm.albums.map(\.name) == ["A", "B"])
}
```

Prefer real implementations over mocks. Only mock at hard boundaries â€” networking, file system, hardware sensors. Use protocol conformances for test doubles rather than mocking frameworks.

For UI testing, use **Maestro** flows to verify critical user journeys.

## Code Style

Use **SwiftFormat** to enforce consistent style. It runs in pre-commit hooks and CI. Follow Swift's standard naming conventions:

- `camelCase` for variables, functions, and parameters
- `PascalCase` for types, protocols, and enum cases
- Boolean properties read naturally: `isLoading`, `hasAccess`, `canEdit`
- Maximum function length: 50 lines â€” extract if longer
